langgrph框架部署本地的agent服务

第一步：环境准备（打好地基）

在开始编码前，我们需要安装必要的依赖。这里我推荐使用 DeepSeek 或 Qwen 的兼容 OpenAI 接口的模型，性价比高且稳定。

# 创建项目目录
mkdir my_langgraph_agent && cd my_langgraph_agent

# 安装核心依赖
pip install langgraph langchain-core langchain-openai python-dotenv

# 如果你打算使用官方的一键部署工具，还需要安装 CLI
pip install -U "langgraph-cli[inmem]"

在项目根目录创建一个 .env文件，填入你的模型 API Key：

OPENAI_API_KEY=sk-your_api_key_here
OPENAI_BASE_URL=https://api.deepseek.com/v1  # 以 DeepSeek 为例，也可以是其他兼容 OpenAI 的地址

第二步：实战——用 LangGraph 组装一个 Agent

LangGraph 的核心哲学非常简单，只有三个概念：状态 (State)、节点 (Nodes) 和 **边 (Edges)**。

  • State（状态）：Agent 的“共享白板”，所有节点都在这块白板上读写数据。
  • Nodes（节点）：具体的“车间工人”，比如“调用大模型”、“执行搜索工具”。
  • Edges（边）：总控“调度中心”，决定流水线下一步该传给谁。

langgraph提供三种上下文方式

  • Configurale:在运行时传入的数据，生命周期，每次运行时
  • AgentState:执行期间可动态修改的数据
  • 长期记忆（存储）：可在对话之间共享的数据

下面我们直接上代码，创建一个 agent.py：

import os
from dotenv import load_dotenv
from typing import TypedDict, List, Union
from langchain_core.messages import HumanMessage, AIMessage, ToolMessage, BaseMessage
from langchain_openai import ChatOpenAI
from langgraph.graph import StateGraph, END
import json

# 1. 定义状态 (State)：我们的共享白板
class AgentState(TypedDict):
    messages: List[BaseMessage] # 用列表存储所有的对话和工具调用记录

# 2. 定义节点 (Nodes)：具体的执行逻辑
# 初始化大模型 (这里使用支持 Tool Calling 的模型)
load_dotenv()
llm = ChatOpenAI(model="deepseek-chat", base_url=os.getenv("OPENAI_BASE_URL"))

# 模拟一个外部工具（比如查询天气的 API）
def fake_weather_tool(city: str) -> str:
    """模拟获取天气信息"""
    return f"{city}今天天气晴朗，温度 25°C，适合出门散步。真实场景这里可以换成 requests 调用实际 API。"

def call_llm(state: AgentState):
    """节点1：调用大模型进行思考"""
    messages = state["messages"]
    response = llm.invoke(messages)
    return {"messages": [response]}

def call_tool(state: AgentState):
    """节点2：执行工具调用"""
    messages = state["messages"]
    last_message = messages[-1]
    
    # 提取模型想要调用的工具和参数
    tool_calls = last_message.tool_calls
    tool_results = []
    
    for tool_call in tool_calls:
        if tool_call["name"] == "get_weather":
            city = tool_call["args"]["city"]
            result = fake_weather_tool(city)
            # 将工具执行结果包装成 ToolMessage 放回白板
            tool_results.append(ToolMessage(content=result, tool_call_id=tool_call["id"]))
            
    return {"messages": tool_results}

# 3. 定义边与条件 (Edges)：路由逻辑
def should_continue(state: AgentState) -> str:
    """判断流水线下一步去哪：如果有工具调用请求，就去执行工具；否则结束。"""
    messages = state["messages"]
    last_message = messages[-1]
    
    # 如果模型发起了工具调用请求
    if hasattr(last_message, 'tool_calls') and last_message.tool_calls:
        return "call_tool"
    # 如果是普通对话，或者工具已执行完毕，就结束本轮思考
    else:
        return END

# 4. 组装流水线 (Graph)
workflow = StateGraph(AgentState)

# 注册节点
workflow.add_node("agent", call_llm)
workflow.add_node("action", call_tool)

# 设置入口
workflow.set_entry_point("agent")

# 添加边和条件路由
workflow.add_conditional_edges("agent", should_continue)
workflow.add_edge("action", "agent") # 工具执行完后，把结果再交给模型总结

# 编译成可运行的图
app = workflow.compile()

# --- 测试一下 ---
if __name__ == "__main__":
    print("🤖 本地 Agent 服务已启动，输入 'exit' 退出。")
    while True:
        user_input = input("You: ")
        if user_input.lower() == 'exit':
            break
        
        # 准备初始状态
        initial_state = {"messages": [HumanMessage(content=user_input)]}
        
        # 流式输出看看内部思考过程
        for event in app.stream(initial_state, stream_mode="values"):
            event["messages"][-1].pretty_print()

(运行 python agent.py即可在终端测试这个 Agent 的交互逻辑)

第三步：部署为本地服务（让外部可以调用）

写完核心逻辑后，我们有两种方式把它变成可供前端或其他服务调用的“本地服务”。

方案一：极简 FastAPI 封装（适合有后端开发经验的玩家）

如果你只是想要一个简单的 HTTP 接口，直接用 FastAPI 包一层是最快最灵活的。

# 安装：pip install fastapi uvicorn
from fastapi import FastAPI
from pydantic import BaseModel
from agent import app as langgraph_app # 导入上面写好的 workflow

app = FastAPI()

class ChatRequest(BaseModel):
    message: str

@app.post("/chat")
def chat_endpoint(req: ChatRequest):
    initial_state = {"messages": [HumanMessage(content=req.message)]}
    result = langgraph_app.invoke(initial_state)
    # 提取最后一条 AI 回复返回
    final_response = result["messages"][-1].content
    return {"response": final_response}

# 启动命令：uvicorn main:app --reload --port 8000

方案二：使用 LangGraph CLI（官方最强推荐 🌟）

如果你想拥有可视化的调试界面（LangGraph Studio）、自动生成的 API 文档以及生产级的并发处理，官方提供的 CLI 工具是绝对的首选。

1. 调整项目结构

确保你的文件夹结构如下：

my_agent_app/
├── agent.py          # 上面写好的核心逻辑
├── requirements.txt  # 依赖列表 (包含 langgraph 等)
└── langgraph.json    # 服务配置文件

2. 创建 langgraph.json

这个文件告诉 LangGraph 怎么找到你的代码：

{
  "graphs": {
    "my_agent": "./agent.py:app" 
  },
  "env": ".env",
  "python_version": "3.11",
  "dependencies": ["."]
}

(注：./agent.py:app指的是 agent.py文件里的 app变量，也就是我们 compile()出来的对象)

3. 一键启动本地服务

在终端进入 my_agent_app目录，执行：

pip install -e .

langgraph dev --allow-blocking

4. 见证奇迹

启动后，终端会输出几个关键链接：

  • 🚀 API 接口: http://127.0.0.1:2024(你的前端可以直接对接这个接口)

  • 🎨 可视化调试 UI (Studio): https://smith.langchain.com/studio/?baseUrl=http://127.0.0.1:2024(直接在你的浏览器里看 Agent 是怎么一步步思考和执行工具的！)

  • 📚 API 文档: http://127.0.0.1:2024/docs(标准的 Swagger 文档，方便联调)

💡 避坑指南与进阶建议

  1. 状态设计是灵魂：State不一定要用 TypedDict，在复杂项目中，继承 langgraph.graph.MessagesState往往更省事，它内置了消息列表的管理。

  2. 持久化记忆：上面的例子每次重启都会失忆。在生产环境中，你需要给 workflow.compile()传入一个 checkpointer（比如基于 SQLite 或 Redis），这样 Agent 就能记住跨线程的历史对话了。

  3. 工具绑定别忘了：在实际代码中，记得在初始化 LLM 后绑定工具，例如 llm_with_tools = llm.bind_tools([tool_definition])，这样模型才知道它能调用什么工具
