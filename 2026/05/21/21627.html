LangChain使用

一、 LangChain Deep Agents 框架详解

Deep Agents 被官方定义为一个 “Batteries-included”（开箱即用）的 Agent Harness（智能体编排框架）。它底层基于 LangChain 和 LangGraph 构建，内置了任务规划、虚拟文件系统、子智能体委派等能力。开发者无需再从零搭建提示词和工具链，只需几行代码就能拉起一个具备长上下文管理能力的复杂 Agent。

  • 🌐 官方文档：https://docs.langchain.com/deepagents
  • 💻 GitHub 仓库：https://github.com/langchain-ai/deepagents

特点：

  • 提供模型统一接口

  • 支持创建Agent/DeepAgent

  • 工具调用，skill -模块开发

  • 记忆管理

自定义方式创建大模型

import json

from typing import Any, Callable, List, Optional, Sequence
from langchain_core.language_models import BaseChatModel, LanguageModelInput
from langchain_core.messages import (
    AIMessage,
    BaseMessage,
    HumanMessage,
    SystemMessage,
    ToolMessage,
)

from langchain_core.output_parsers.openai_tools import (
    make_invalid_tool_call,
    parse_tool_call,

)

from langchain_core.outputs import ChatGeneration, ChatResult
from langchain_core.runnables import Runnable
from langchain_core.tools import BaseTool
from langchain_core.utils.function_calling import convert_to_openai_tool
from zai import ZhipuAiClient
def _lc_tool_call_to_openai(tool_call: dict) -> dict:

    return {
        "type": "function",
        "id": tool_call["id"],
        "function": {
            "name": tool_call["name"],
            "arguments": json.dumps(tool_call["args"], ensure_ascii=False),
        },
    }
def _format_tool_choice(
    tool_choice: dict | str | bool | None,
    tool_names: list[str],
) -> str | dict | None:
    if not tool_choice:

        return None

    if isinstance(tool_choice, str):

        if tool_choice in tool_names:

            return {"type": "function", "function": {"name": tool_choice}}

        if tool_choice in ("any", "required"):

            return "auto"

        return tool_choice

    if isinstance(tool_choice, bool):

        return "auto"

    return tool_choice





class ZhipuAIChatModel(BaseChatModel):

    """LangChain chat model wrapper for Zhipu AI (zai SDK) with OpenAI-style tool calling."""



    client: ZhipuAiClient

    model_name: str



    def __init__(self, api_key: str, model_name: str = "glm-4.7-flash", **kwargs):

        client = ZhipuAiClient(api_key=api_key)

        super().__init__(client=client, model_name=model_name, **kwargs)



    def bind_tools(

        self,

        tools: Sequence[dict[str, Any] | type | Callable | BaseTool],

        *,

        tool_choice: dict | str | bool | None = None,

        **kwargs: Any,

    ) -> Runnable[LanguageModelInput, AIMessage]:

        formatted_tools = [convert_to_openai_tool(tool) for tool in tools]

        tool_names = [

            t["function"]["name"]

            for t in formatted_tools

            if "function" in t and "name" in t["function"]

        ]

        formatted_choice = _format_tool_choice(tool_choice, tool_names)

        bind_kwargs: dict[str, Any] = {"tools": formatted_tools}

        if formatted_choice is not None:

            bind_kwargs["tool_choice"] = formatted_choice

        bind_kwargs.update(kwargs)

        return self.bind(**bind_kwargs)



    def _convert_messages(self, messages: List[BaseMessage]) -> list[dict]:

        converted: list[dict] = []

        for m in messages:

            if isinstance(m, HumanMessage):

                converted.append({"role": "user", "content": m.content or ""})

            elif isinstance(m, SystemMessage):

                converted.append({"role": "system", "content": m.content or ""})

            elif isinstance(m, AIMessage):

                msg: dict[str, Any] = {

                    "role": "assistant",

                    "content": m.content or None,

                }

                if m.tool_calls:

                    msg["tool_calls"] = [

                        _lc_tool_call_to_openai(tc) for tc in m.tool_calls

                    ]

                    if not msg["content"]:

                        msg["content"] = None

                converted.append(msg)

            elif isinstance(m, ToolMessage):

                converted.append(

                    {

                        "role": "tool",

                        "content": m.content,

                        "tool_call_id": m.tool_call_id,

                    }

                )

            else:

                converted.append({"role": "user", "content": str(m.content)})

        return converted



    def _response_to_aimessage(self, response_message: Any) -> AIMessage:

        content = response_message.content or ""

        if not getattr(response_message, "tool_calls", None):

            return AIMessage(content=content)



        tool_calls = []

        invalid_tool_calls = []

        for tc in response_message.tool_calls:

            raw = {

                "id": tc.id,

                "type": tc.type,
                "function": {
                    "name": tc.function.name,
                    "arguments": tc.function.arguments,
                },
            }
            try:
                parsed = parse_tool_call(raw, return_id=True)
                if parsed:
                    tool_calls.append(parsed)
            except Exception as e:
                invalid_tool_calls.append(make_invalid_tool_call(raw, str(e)))
        return AIMessage(
            content=content,
            tool_calls=tool_calls,
            invalid_tool_calls=invalid_tool_calls,
        )
    def _generate(
        self,
        messages: List[BaseMessage],
        stop: Optional[List[str]] = None,
        run_manager: Optional[Any] = None,
        **kwargs: Any,
    ) -> ChatResult:
        api_kwargs: dict[str, Any] = {
            "model": self.model_name,
            "messages": self._convert_messages(messages),
            "max_tokens": kwargs.get("max_tokens", 65536),
            "temperature": kwargs.get("temperature", 1.0),
        }
        if stop:
            api_kwargs["stop"] = stop
        if "tools" in kwargs:
            api_kwargs["tools"] = kwargs["tools"]
            api_kwargs["thinking"] = {"type": "disabled"}
        else:
            api_kwargs["thinking"] = kwargs.get("thinking", {"type": "enabled"})

        if "tool_choice" in kwargs:
            api_kwargs["tool_choice"] = kwargs["tool_choice"]
        response = self.client.chat.completions.create(**api_kwargs)
        message = self._response_to_aimessage(response.choices[0].message)
        return ChatResult(generations=[ChatGeneration(message=message)])
    @property
    def _llm_type(self) -> str:
        return "zhipu-ai"

核心能力（为什么要用它？）

传统的 Agent 往往只能处理简单的“一问一答”或单一工具调用，而 Deep Agents 专为解决复杂、长周期任务设计：

  1. 📝 自主规划与任务分解：内置 write_todos工具，Agent 收到复杂需求后会先拆解成 Checklist，步步为营。
  2. 📂 虚拟文件系统与上下文管理：自带 read_file, write_file等工具。面对海量搜索结果或大段代码，Agent 能将其暂存至文件系统中，防止大模型“内存溢出”（Context Window 超限），并支持自动摘要压缩。
  3. **🪄 子智能体委派 (Sub-agents)**：遇到特别专精的子任务，主 Agent 可以生成独立的“子 Agent”去处理，实现完美的上下文隔离。
  4. 🖥️ 沙盒化终端执行：在配置沙盒后端后，Agent 可获得 execute权限运行 Shell 命令，真正实现代码测试、环境部署等自动化。
  5. **🛡️ 人工干预机制 (HITL)**：针对高风险操作（如删库、大额退款），可配置人工审批工作流

环境安装步骤 (使用 uv)

现在的 LangChain 采用了高度模块化的设计，核心功能在 langchain-core，具体的模型接入（如 OpenAI、DeepSeek、通义千问等）和高级功能则在各自的扩展包中。

打开你的 Win11 终端，在项目虚拟环境中执行以下命令：

# 1. 安装 LangChain 核心库以及常用的社区库
uv pip install langchain-core langchain langchain-community

# 2. 安装你想要使用的大模型提供者 SDK (以 OpenAI 兼容接口为例，DeepSeek/通义千问等均通用)
uv pip install langchain-openai

uv通过 Pip 安装（适合已配置好 Python 环境的用户）

如果你机器上已经有 Python 环境，并且懒得折腾，直接用 pip 一键安装即可：

pip install uv

uv –version

为了方便你直观对比，我把最常用的场景列成了表格：

场景 传统 Pip 命令 ⚡️ Uv 极速命令 额外亮点说明

安装单个包 pip install requests uv pip install requests 自动利用全局缓存，速度飞起

  1. 环境同步（Synchronize）

如果你想让当前的 Python 虚拟环境与 requirements.txt文件强制保持完全一致（多退少补），uv提供了一个非常硬核的命令。它会自动安装文件里有但环境中缺失的包，并无情卸载环境中存在但文件里没有的冗余包：

uv pip sync requirements.txt
  1. 依赖锁定（Lock / Compile）

在团队协作中，为了保证大家用的依赖版本绝对一致，uv可以读取一个只有顶层依赖的 requirements.in文件，快速计算出所有嵌套依赖的确切版本，并生成锁定的 requirements.txt（支持 --universal跨平台解析）：

uv pip compile requirements.in --output-file requirements.txt
  1. 免激活运行脚本（Run）

这是 uv非常讨喜的一个现代特性。你可以直接在项目根目录运行 Python 脚本，如果脚本带有内联的依赖元数据（比如声明了需要 requests库），uv会在临时的虚拟环境里自动帮你装上依赖并立刻执行，连手动激活环境都省了：

uv run main.py

指定 Python 版本创建（超实用）

假设你当前系统装的是 Python 3.12，但这老项目非得用 Python 3.9 才能跑，你可以直接指定版本：

uv venv --python 3.9

如果系统里没有 3.9，uv会在命令行里提示你，你只需按一下 y确认，它就会自动下载安装，接着无缝创建出 3.9 的虚拟环境。

  1. 指定虚拟环境名称

默认生成的文件夹名叫 .venv，如果你不喜欢这个名字（比如想叫 my_env），可以加个参数：

uv venv my_env

二、 不同虚拟环境的跳转 (Activate & Deactivate)

“跳转”在官方术语里叫激活（Activate）。在 Win11 的终端里，激活环境的命令和以前使用 venv或 virtualenv时完全一样。

  1. 进入（激活）虚拟环境

根据你使用的终端不同，命令略有区别：

  • 如果你用 PowerShell：

    .\.venv\Scripts\Activate.ps1

    (注意开头的点和反斜杠)

  • 如果你用 Command Prompt (CMD) 或 Git Bash：

    .venv\Scripts\activate

激活成功后，你会明显看到终端提示符的最前面多了一个括号，里面显示环境名称，比如 (.venv) C:\MyProject>。这就代表你已经成功“走进”了这个小房间，接下来运行 python或 pip都是在这个隔离环境里生效。

  1. 退出（跳转回全局环境）

无论你在哪个目录下，只要你想跳出当前的虚拟环境，回到系统的全局 Python 环境，只需简单输入：

deactivate

输入后，你会发现终端前面的 (.venv)消失了，这就代表你成功“跳出”了该虚拟环境

剔除无关依赖（重要）

系统全局环境（base）中往往包含 pip, setuptools, wheel等基础包，或者是某些 IDE 偷偷安装的插件。你可以使用 -X参数（uv）或 --exclude参数（pip）来过滤它们，保持文件的纯净：

# 使用 uv 剔除特定包
uv pip freeze -X pip -X setuptools -X wheel > requirements.txt

# 或者使用 pip 剔除
pip freeze | findstr /V "pip setuptools wheel" > requirements.txt

常用函数与核心 API 详解

Deep Agents 秉持“约定大于配置”的原则，核心入口只有一个，但通过参数提供了极高的定制化自由度。

1. 核心构造函数：create_deep_agent()

这是创建 Agent 的唯一主函数，其最常用参数如下：

参数名 类型 必填 说明
model str或 LangChain Model 否 指定底层大模型。推荐使用支持 Tool Calling 的前沿模型（如 openai:gpt-4o, anthropic:claude-3-5-sonnet）。
tools List[Callable] 否 注入自定义工具函数列表（只需是标准的 Python 函数，加上类型提示和 DocString 即可）。
system_prompt str 否 覆盖默认的 System Prompt，赋予 Agent 特定的角色设定或工作流指引。
middleware List[AgentMiddleware] 否 注入自定义中间件，可用于高级的 Prompt 拼接或生命周期钩子（Lifecycle hooks）。
subagents List[CompiledSubAgent] 否 预定义可供主 Agent 调用的子 Agent 列表。
backend FilesystemBackend 否 配置文件系统后端。默认为内存态，可设置为本地磁盘同步或 LangGraph 持久化存储。

2. 内置工具集（开箱即用）

只要你使用 create_deep_agent，以下工具会自动挂载到 Agent 身上，无需手动编写：

  • **write_todos/ read_todos**：任务白板，让 Agent 学会做项目管理。

  • **read_file/ write_file/ edit_file**：文件读写，解决长文本存储问题。

  • **glob/ grep**：代码库检索，让 Agent 具备类似程序员的全局搜索能力。

  • **execute**：在沙盒环境中执行系统命令（需配合特定 Backend 开启）。

进阶玩法

  1. 🌊 流式输出 (Streaming)

    Deep Agents 底层基于 LangGraph，原生支持流式事件。只需将 .invoke()替换为 .stream()，即可在终端实时看到 Agent 的思考过程和工具调用详情。

  2. 🧩 MCP (Model Context Protocol) 集成

    可以通过 langchain-mcp-adapters库，一行代码接入任何支持 MCP 协议的标准化工具服务器，瞬间拓展 Agent 能力边界（如连接本地数据库、Slack、Notion 等）。

  3. 💻 Deep Agents CLI（终端利器）

    该框架还附带了一个开箱即用的 Terminal UI 编码助手（类似 Claude Code 或 Cursor 终端版）。安装后直接在命令行输入 deepagents，就能拥有一个带富文本流式输出、联网搜索和代码执行能力的终端 Copilot

DeepAgent 项目：就像是给这个员工配上了一整套办公软件（规划本、文件夹、能委派任务的通讯录），让他能独立自主地完成复杂的长周期项目。

多模态 RAG：就像是给这个员工接入了一个庞大的多媒体图书馆（里面有文字、图片、视频），并教会他如何快速查资料来辅助解答老板的疑问。

微调项目 (Fine-tuning)：这不是教员工怎么用工具，而是直接让他回炉重造，去攻读一个特定的专业学位（比如法学博士或医学博士），使他本身的脑子就具备某一领域的专家级直觉

一、 DeepAgent 项目（以 LangChain Deep Agents 为例）

DeepAgent 是一类专注于解决复杂、多步骤、长周期任务的智能体（Agent）框架。以 LangChain 最新开源的 Deep Agents 为例，它不再仅仅是简单的“一问一答”或单次工具调用，而是一个自带“方法论”的智能体脚手架（Agent Harness）。

1. 核心特点

  • 自带“任务拆解”能力（规划）：内置 TodoList 工具，接到复杂指令后，它会自动将大任务拆解为具体的待办事项，步步为营。

  • 虚拟文件系统（上下文管理）：这是它最亮眼的设计。它能把大量的搜索结果或长文本存进“虚拟文件”里，避免大模型“内存溢出”（Context Window 超限），极大降低了 Token 消耗和成本。

  • 子智能体委派（Multi-Agent）：遇到特别专精的子任务，主 Agent 可以像领导一样，把任务派发给临时生成的“子 Agent”去独立完成，隔离上下文，互不干扰。

  • 高可控与可干预：支持人工介入审批（HITL），对于高风险操作可以设置人工确认；同时支持中途恢复，断点续跑。

2. 主要用途

  • 深度自动化研究：给定一个主题，自动上网搜索、阅读多篇网页、交叉验证信息，最后输出一份详实的 Markdown 调研报告。

  • 全自动编程与调试：接收需求文档，自动拆解开发任务，编写代码，甚至在沙盒环境中运行测试，根据报错自动修复（Self-Debugging）。

  • 复杂的客户旅程模拟：模拟用户进行多轮对话，执行一连串的系统操作。

3. 实现步骤

以使用 LangChain Deep Agents 为例：

# 1. 安装依赖
pip install deepagents langchain-openai

# 2. 编写极简代码
from deepagents import create_deep_agent
from langchain_openai import ChatOpenAI

# 3. 配置模型并挂载工具（甚至可以留空，使用内置的规划与文件工具）
llm = ChatOpenAI(model="gpt-4o")
agent = create_deep_agent(model=llm, tools=[])

# 4. 抛出复杂任务，让它自行拆解和执行
result = agent.invoke({"messages": [{"role": "user", "content": "帮我调研一下竞品最近的营销策略，输出一份对比报告"}]})
print(result["messages"][-1].content)

二、 多模态 RAG（Multimodal Retrieval-Augmented Generation）

传统 RAG 主要处理纯文本，而多模态 RAG 将检索和生成的边界拓展到了图片、表格、音频甚至视频。它让大模型能够“看图说话”，利用外部的多媒体知识库来回答问题时，不仅依据文字，还能看懂图表和截图。

1. 核心特点

  • 跨模态对齐（Cross-modal Alignment）：通过如 CLIP 这样的多模态编码器，将图片和文本映射到同一个高维向量空间中，实现“用文字搜图片”或“用图片搜文字”。

  • 多路召回机制：查询时，系统会并行发起对文本向量、图片向量、表格向量的检索，最后合并去重交由大模型生成。

  • 证据溯源能力：生成的答案可以精准追溯到具体的图文块，减少了大模型“自由发挥”（幻觉）的空间。

2. 主要用途

  • 企业级智能知识库：员工上传含有大量架构图、数据报表的 PDF 产品手册，系统能直接根据图纸回答配置问题。

  • 视觉问答（VQA）与电商导购：用户拍一张衣服的照片问“这件衣服配什么裤子好看”，系统检索相似的穿搭图文并给出建议。

  • 医疗影像报告生成：结合患者的 X 光片（图）和既往病历（文），自动生成初步诊断报告。

3. 实现步骤

构建一个多模态 RAG 通常包含以下几个核心阶段：

  1. 多模态数据预处理：使用 OCR 提取文档中的文字，使用视觉模型（如 ViT）提取图片特征，使用 ASR 转录音频。

  2. 统一向量化与存储：使用多模态 Embedding 模型（如 CLIP）将上述不同模态的数据转化为向量，一并存入支持混合检索的向量数据库（如 Milvus、FAISS）。

  3. 多路召回与重排（Rerank）：用户提问后，并行在文本、图片等索引中检索 Top-K 个相似项，然后通过一个 Reranker 模型统一打分，筛选出最相关的图文证据。

  4. 多模态大模型生成：将原始问题连同检索到的文本片段和图片 URL 一起喂给多模态大模型（如 GPT-4V、Gemini Pro），生成最终回答。

三、 大模型微调项目（Fine-tuning）

微调是指在保留了大模型通用能力的基础上，使用特定领域的高质量数据集对模型的部分或全部参数进行二次训练。如果说 RAG 是给模型发了一本“考试时的参考书”，那微调就是利用闭关修炼，把知识彻底“刻进 DNA 里”。

1. 核心特点

  • 领域深度适配：能够让通用模型迅速掌握生涩的行业术语、特定的代码规范或品牌的特有文风。

  • 降低幻觉与提升遵从度：通过大量的指令-响应对（Instruction Tuning），模型会更严格地按照要求的格式和逻辑输出，减少乱扯的概率。

  • 参数高效微调（PEFT）技术普及：借助 LoRA 或 QLoRA 等技术，现在只需训练极小部分新增参数（不到全量的 1%），单张消费级显卡（如 RTX 3090）就能微调几十亿参数的大模型，大幅降低了门槛。

2. 主要用途

  • 垂直领域专家模型：使用医学论文和病例微调，得到医疗问诊专家；使用法条和判决书微调，得到法律服务助手。

  • 特定代码库的开发助理：用公司内部庞大的私有代码库微调，让模型补齐的代码完美契合内部的函数命名和架构习惯。

  • 角色扮演与特定语气生成：微调一个只会用“互联网黑话”写周报的模型，或者一个极具亲和力的心理咨询模型。

3. 实现步骤

一个标准的企业级微调流程如下：

  1. 需求拆解与数据准备：明确业务目标，收集几千到几万条高质量的“输入-输出”配对数据。数据质量直接决定微调成败，通常需要大量的人工清洗和标注。

  2. 基座模型选型：根据算力预算选择底座（如 Qwen、Llama、ChatGLM）。

  3. 训练策略配置：决定是全参数微调（算力要求极高）还是 LoRA 微调。设置学习率（Learning Rate）、批次大小（Batch Size）等超参数。

  4. 启动训练与监控：在 GPU 集群上启动训练，实时监控 Loss 曲线，防止过拟合。

  5. 评估与部署：使用测试集验证微调效果，导出模型权重，通过推理加速框架（如 vLLM）部署上线提供服务。

💡 核心区别总结

为了帮您在不同场景下做技术选型，我们可以看下它们的核心区别：

维度 DeepAgent 多模态 RAG 微调 (Fine-tuning)
解决痛点 解决“怎么做”的问题（任务执行与自动化） 解决“不知道”的问题（拓展知识边界与查缺补漏） 解决“不专业/不会做”的问题（改变模型内在能力与风格）
知识新鲜度 可通过工具实时获取最新信息 依赖知识库更新，可做到准实时/实时 固化在模型参数中，更新知识需要重新训练
运维成本 需维护工具链路、沙盒环境与中间件 需维护向量数据库、Embedding 模型及检索链路 训练成本高，但推理成本通常低于其他两者
典型代表 LangChain Deep Agents, AutoGen LlamaIndex Multimodal, LangChain + Milvus PEFT/LoRA, Hugging Face SFTTrainer

最佳实践往往是“组合拳”：

用微调打底，塑造模型的基础性格和专业术语理解能力；

在此之上构建多模态 RAG 层，提供企业最新的私有图文数据支持；

最外层套上 DeepAgent 框架，赋予其规划、拆解和调用工具去解决实际业务问题的能力
