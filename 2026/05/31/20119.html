一、Imatest

🌐 官网链接

https://www.imatest.com/

快速入门文档：https://www.imatest.com/docs/instructions-getting-started/

📌 定位与用途

Imatest 是影像行业应用最广的图像质量客观分析软件之一，已有 20+ 年历史。核心思路是：拍摄标准测试图卡（Test Chart）→ 用软件自动ROI识别→ 输出量化指标。用于：

  • 研发阶段：评估相机/镜头/ISP算法性能，指导调优
  • 竞品对比：不同相机在相同条件下做可复现对标
  • 产线质检：自动化批量检测，确保出厂一致性

🔧 主要功能（可测参数）

类别 典型指标
清晰度 MTF / SFR（斜边缘法，ISO 12233）
色彩 色准 ΔΕ、色域、白平衡
色调/动态 OECF、色调响应曲线、动态范围
噪声 SNR、噪声频谱、Shannon信息容量
几何光学 畸变 Distortion、色散（LCA/CA）、视场角 FoV
均匀性 Shading / Luma uniformity
杂光 Flare / 鬼影 / 眩光
缺陷 Blemish、坏点、Artifacts / Moiré

最核心的两个模块是 SFRplus 和 eSFR ISO ——拍一张图就能一次性出大量指标。

▶️ 基本使用步骤

① 确定要测的指标（锐度？色彩？畸变？动态范围？）
② 选对应图卡（ISO 12233 / ColorChecker / 灰阶楔形…）
③ 在可控光照下拍摄图卡（按模块要求取景/填充画面）
④ 打开 Imatest → 选择对应分析模块 → 导入图像
⑤ 软件自动ROI检测 → 读取图表数值 → 导出 CSV/JSON 报告
⑥ 对比基线/竞品/规格要求，判断 Pass/Fail
  • 新手可以用 GitHub 上的 sample images 先跑通流程
  • 首次运行自带 14天免费试用，之后需购买 License 激活
  • 也支持 命令行/API 集成到工厂自动化产线

一、音频 3A：AEC · ANS · AGC

3A = 三类音频前处理算法的总称，几乎是所有实时语音系统（VoIP、视频会议、语音助手、车载通话）的”前置净化器”。处理链路顺序是 AEC → ANS → AGC（先消回声，再抑噪声，最后管音量）。

1. AEC — Acoustic Echo Cancellation（声学回声消除）

问题从哪来？

假设你手机开免提跟 B 通话：

B的声音 → 你手机扬声器播放 → 通过空气传播 + 墙面反射 
         → 被你手机麦克风拾取 → 混进你的说话声 → 发回给B
         → B 听到了自己的回声（甚至啸叫）

关键点：这不是电学回流，而是声学路径的回声——扬声器播放的信号经房间冲激响应（Room Impulse Response, RIR）染色后回到麦克风。

ANS — Automatic Noise Suppression（自动噪声抑制）

也叫 Noise Reduction / Speech Enhancement。目标是：保留语音，压制一切非语音（空调嗡嗡声、风扇、键盘、交通、风噪）

用途场景

  • 任何”麦克风在真实世界”的场景：耳机通话、会议麦克风阵列、直播麦克风
  • 语音识别前端（ASR 对 SNR 极敏感，ANS 是标配）

3. AGC — Automatic Gain Control（自动增益控制）

问题

同一个会议室里：有人凑到麦跟前（爆音削波），有人靠后坐（听不见）；同一个人在说话时音量也会自然起伏。

用途场景

  • 语音通信（让双方音量体感”差不多大”）

  • 录音（防止过载 + 避免太弱需要后期猛拉）

3A 的处理顺序为什么是 AEC → ANS → AGC？

麦克风原始信号
    │
    ▼
 ┌─────────┐   回声是"确定性可参考"的，必须先消，
 │  AEC    │   不然回声混进信号后，ANS会把回声当噪声
 │(先对消) │   模型破坏，AGC也会跟着误判电平
 └────┬────┘
      ▼
 ┌─────────┐   噪底压下去，信号主体干净了
 │  ANS    │
 └────┬────┘
      ▼
 ┌─────────┐   最后才调音量——避免 AGC 放大噪声、
 │  AGC    │   或因为回声脉冲导致增益疯狂振荡
 └────┬────┘
      ▼  干净输出 → 编码 → 网络

交换顺序会出问题：ANS 在 AEC 前面 → ANS 可能把回声伪当成”稳态噪声”吃掉一部分但留残影，反而更难收敛；AGC 在 AEC 前面 → 回声幅度被缩放后，自适应滤波器参考信号对不上。

一、视频客观质量评价：PSNR / SSIM / VMAF

三者都属于 Full-Reference（全参考） 指标——必须有原始无损/准无损参考视频，逐帧比对失真视频。

1. PSNR — Peak Signal-to-Noise Ratio（峰值信噪比）

原理

逐像素算均方误差，再转成对数

觉：它只回答”像素值差了多少”，本质是 ℓ₂ 距离 的变体。不理解人眼，不理解结构、纹理、掩蔽效应。

主要用途

场景 适合吗 为什么
编码器回归测试（”这次改动有没有炸”） ✅ 极快、稳定、可复现
压缩损伤粗筛 △ 数值高≠好看，但对”崩了”很敏感
替代人工主观评分 ❌ 经典反例：模糊后 PSNR 可能还高，但人觉得糊

经验阈值参考

PSNR (Y) 主观感受
>40 dB 可接受～好
>45–50 dB 优秀（接近透明）
< 30 dB 明显损伤

⚠️ 这只是经验区间，不同内容差异极大——这就是需要 SSIM/VMAF 的原因

使用步骤（FFmpeg，最常用）

# 确认 ffmpeg 编译了 libvmaf（大多数发行版自带）
ffmpeg -i reference.mp4 -i distorted.mp4 \
  -lavfi psnr="stats_file=psnr.log" \
  -f null -

输出示例：PSNR y:38.52 u:42.15 v:43.08 average:39.12→ 看 average行

2. SSIM — Structural Similarity Index Measure（结构相似性）

原理

Wang & Bovik 2004 提出，核心主张：人眼对”结构信息”的敏感度远高于绝对像素误差

为什么比 PSNR 好：对模糊、平滑类失真更敏感；能生成 质量热力图（哪块区域掉分一目了然）。但仍对色彩失真/高频纹理丢失不够敏感，且假设对齐完美（无时移）
主要用途
编码器调参时的结构保真度标尺
和 PSNR 一起跑，交叉验证（PSNR 高但 SSIM 低 → 可能糊 / 细节抹平）
生成 per-frame SSIM 曲线看 GOP 周期性波动
使用步骤（FFmpeg）
ffmpeg -i reference.mp4 -i distorted.mp4
-lavfi ssim=”stats_file=ssim.log”
-f null -
输出：SSIM All:0.9601（越靠近 1 越好）
3. VMAF — Video Multi-Method Assessment Fusion
Netflix 出品，目前业界评估”主观观感”最靠谱的单一指标。
官网/源码：https://github.com/Netflix/vmaf
Netflix 技术博客：VMAF: The Journey Continues（搜索 “Netflix Technology Blog VMAF”）
为什么需要 VMAF
经典例子：两帧 PSNR 都是 31dB，但一张是纹理细节被抹平（狐狸毛发糊成一片），另一张只是轻微噪声——PSNR 分辨不出，但人一眼看出前者糟糕得多。VMAF 就是用 ML 把”人眼怎么评”学出来。
原理（三层架构）
参考帧 R + 失真帧 D
│
├── VIF (Visual Information Fidelity) → “保留了多少信息”
├── ADM (Additive Distortion Measure / DLM) → “结构破坏/增量失真”
├── Motion特征 → “运动强度（影响时域掩蔽）”
│
▼
特征向量 → SVM回归器(或神经网络) → 融合 → 0100 分
VIF：基于自然场景统计（NSS）模型，量化有多少原始内容信息被保留
ADM/DLM：对增量式失真（ringing、blur halo、噪声残留）敏感
Motion：动态场景下运动会影响人眼容忍度（静止时瑕疵更显眼）
最终用 SVM regression 把多个特征映射到 MOS 尺度 → 输出 0–100（越高越好）
默认模型对应观看条件：1080p 模型 ≈ 3倍屏幕高度距离看 1080p TV；4K 模型 ≈ 1.5倍距离看 4K TV。
分数解读
VMAF

主观感受

95–100

近乎透明

80–95

优秀（Excellent）

60–80

良好（Good）

40–60

一般（Fair）

< 40

差
主要用途
编码器选型 / 码率阶梯（ladder）设计：”同样的 3Mbps，x265 vs AV1 谁分高？”
per-title encoding：Netflix 用它做 per-chunk 最优码率决策
视觉质量门禁：CI/CD 里跑 VMAF ≥ 阈值 才让合并
使用步骤
方式 A：FFmpeg 内置 libvmaf（最方便）

① 安装（Ubuntu 例）

sudo apt install ffmpeg
pip install vmaf # 装模型文件到标准路径

② 跑 VMAF（两路输入必须是同分辨率/同帧数/对齐好的）

ffmpeg -i distorted.mp4 -i reference.mp4
-lavfi libvmaf=”model_path=/usr/share/model/vmaf_v0.6.1.json:log_path=vmaf.log:log_fmt=log”
-f null -

③ 一次跑 PSNR+SSIM+VMAF

ffmpeg -i reference.mp4 -i distorted.mp4
-lavfi “[0:v][1:v]libvmaf=model=version=vmaf_v0.6.1:log_fmt=json:log_path=out.json:psnr=1:ssim=1”
-f null -
注意：较新版本 FFmpeg 的 libvmaf 直接用 .json模型文件。模型从 GitHub releases 下载：vmaf_v0.6.1.json、vmaf_4k_v0.6.1.json等
方式 B：从源码跑 Python pipeline
git clone https://github.com/Netflix/vmaf.git
cd vmaf && python setup.py install

把 mp4 解码为 yuv420p 后调用 run_vmaf

PSNR / SSIM / VMAF 三者关系一句话
PSNR 告诉你”变了多大”，SSIM 告诉你”结构还在不在”，VMAF 告诉你”人看着行不行”。 工程中通常 三个一起打，互相兜底

二、音频客观质量评价：PESQ / POLQA / STOI

三者针对的是语音（speech），不是音乐。音乐质量要用 ViSQOL / PEAQ / 主观听音。

1. PESQ — Perceptual Evaluation of Speech Quality

标准：ITU-T P.862 系列（P.862.1 / P.862.2 / P.862.3）

官网/权威入口：https://www.itu.int/rec/T-REC-P.862

核心思想：模拟人耳的频带掩蔽 + 响度非线性 + 时间整合窗口，把”物理差异”转成”感知差异”

局限

  • 仅支持 窄band 8kHz / wideband 16kHz 采样——超宽带（48kHz 全带）不行

  • 对现代 VoIP 的 PLC（packet loss concealment）、时变延迟、噪声鲁棒性不够

  • 所以对高清语音/VoLTE/Vo5G 场景会被 POLQA 取代

使用步骤（最常见：开源 pesq工具）

# ① 下载编译（Python wrapper 或 C 原版）
pip install pesq

# ② 用 Python
from pesq import pesq
import soundfile as sf

ref, sr_ref = sf.read("reference.wav")
deg, sr_deg = sf.read("degraded.wav")
# sr 必须为 8000 或 16000
score = pesq(sr_ref, ref, deg, 'wb')  # 'nb' or 'wb'
print("PESQ MOS-LQO:", score)

# ③ 或命令行 C 实现（ITU参考代码 / github.com/dennisguse/pesq）
pesq -wb reference.wav degraded.wav

前提：两段音频必须时间对齐（同起点、同长度或带已知delay），采样率 8k/16k，单声道，16-bit PCM WAV。

2. POLQA — Perceptual Objective Listening Quality Analysis

标准：ITU-T P.863（2004 初版 → 2011 正式确立，持续修订）

原理（与 PESQ 的同与异）

整体框架类似（对齐 → 听觉建模 → 扰动聚合 → MOS），但关键升级：

维度 PESQ (P.862) POLQA (P.863)
带宽 8k / 16k（窄/宽带） 8k / 16k / 48k（超宽带全带）
模型 较早期感知模型 更新的 Time-Frequency Auditory Model，对瞬态/PLC/电平突变更鲁棒
适用场景 传统 PSTN / 早期 VoIP HD Voice / VoLTE / 4G / 5G / 蓝牙 / EVS / Opus
输出 MOS-LQO ≈ 1–4.5 MOS-LQO 1–5（更接近真实主观 MOS）

准确性上，POLQA 对现代 codec（EVS、Opus、LC3）和声学路径（手扬/免提/回声后）都显著优于 PESQ。

⚠️ 官网与使用方式——重要

POLQA 不是免费开源。参考实现/商业运行时由 OPTICOM GmbH（现多半通过 partners）授权分发，也集成在测试仪表里：

  • Audio Precision APx → 内置 POLQA/PESQ 选件（图形化，最常用）

  • HEAD acoustics ACQUA 平台（通信声学顶级方案）

  • Rohde & Schwarz / Keysight 语音质量测试仪

  • 也有独立 POLQA 软件授权（OPTICOM 官网查询）

如果你只是做内部 codec 评估但不买 POLQA 授权，工程上常用 PESQ 做趋势 + 真听主观确认 或 Deep Learning MOS 模型（NISQA / MOSNet）做近似。

3. STOI — Short-Time Objective Intelligibility（短时客观可懂度）

STOI 测的不是”音质好不好听”，而是”话还能不能听懂”。

原理

STOI 走完全不同的路线——可懂度（intelligibility）视角：

心洞察：语音可懂度的瓶颈在调制包络（envelope），不在精细波形。只要各频带的能量起伏轮廓保留住，哪怕有点失真也能听懂。

  • 输出 0 ~ 1，越高越”能听懂”

  • 典型阈值感受：> 0.75–0.80 ≈ 可懂度良好；< 0.50 ≈ 严重听不清风险

主要用途

场景 为什么 STOI
助听器/降噪算法评估 降噪太狠→包络被毁→STOI掉→虽然安静但不懂
3A（ANS/NS）算法验收 PSNR可能好但把辅音擦音抹了 → STOI立刻报警
丢包/PLC策略评估 看”断了但人能不能连起来”

使用步骤（Python，开源）

pip install stoi  # 或直接从 pystoi
from pystoi.stoi import stoi
import soundfile as sf

ref, sr = sf.read("reference.wav")
deg, _  = sf.read("degraded.wav")
# 统一单声道、同长度、同sr（通常16k）
score = stoi(ref, deg, sr, extended=False)
print("STOI:", score)   # 0~1

PESQ / POLQA / STOI 一句话定位

指标 问的问题 尺度 你能轻易跑吗
PESQ “听起来像原音吗”（品质） MOS-LQO 1–4.5 ✅ 开源 pip/命令行
POLQA 同上，但覆盖全带+现代codec+声学 MOS-LQO 1–5 ❌ 需商业授权（APx/HEAD/OPTICOM）
STOI “话还能听懂吗”（可懂度） 0–1 ✅ 开源 pystoi
