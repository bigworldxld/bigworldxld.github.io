1. MediaInfo (CT机：透视视频底层的每一个参数)

官方网站： https://mediaarea.net/en/MediaInfo

MediaInfo 分为命令行版 (CLI) 和 **图形界面版 (GUI)**，两者可以和谐共存。

Windows 安装步骤

  1. 进入官网的 Download页面。

  2. 您会看到两个主要的下载选项：

    • **Graphical (GUI)**：往下翻，选择 MediaInfo_GUI_*.exe（这是带窗口的可视化版本，新手首选）。

    • **Command Line (CLI)**：往下翻，选择 MediaInfo_CLI_*.exe（这是纯命令行版本）。

  3. 双击下载好的 .exe文件，一路点击 Next（下一步），使用默认设置完成安装即可。

macOS 安装步骤

  1. 前往官网的 Download页面。
  2. 根据您的芯片类型下载对应的 .dmg安装包（Intel 芯片选 x86-64，Apple Silicon 选 ARM64）。
  3. 打开 .dmg文件，将 MediaInfo.app图标拖拽到旁边的 Applications（应用程序）文件夹中。
  4. (CLI额外步骤)*：要将命令行工具集成到终端，需在终端执行：sudo cp /Applications/MediaInfo.app/Contents/MacOS/MediaInfo /usr/local/bin/。

Linux 安装步骤

以 Ubuntu/Debian 为例，打开终端直接输入以下命令：

sudo apt-get update
sudo apt-get install mediainfo   # 安装命令行版
sudo apt-get install mediainfo-gui # 安装图形界面版

2. FFmpeg Batch Converter (流水线：批量音视频转码与压制工厂)

官方网站： http://ffmpeg-batch.sourceforge.io/

⚠️ 提前避坑： 这是一款基于 Windows 开发的专用工具，界面非常友好，支持拖拽和进度条，能最大化利用您的 CPU 进行多文件并发转码。虽然通过网络资料得知它可以通过 Wine 在 Linux 上兼容运行，但在 macOS 上基本无法使用。

Windows 安装步骤

  1. 打开官方网站，点击大大的蓝色按钮下载最新的安装包（通常是 .msi或 .exe格式）。

  2. 双击运行安装程序。

    • 如果跳出安全警告，点击“更多信息” -> “仍要运行”。

    • 在安装向导中，一路点击 Next，直到出现 Install按钮，点击开始安装。

  3. 安装完成后，在开始菜单搜索 FFmpeg Batch即可打开。

  4. 首次使用必看：软件是基于 FFmpeg 的，但它通常会自动在后台为您下载并配置好 FFmpeg 组件。如果界面提示需要 FFmpeg，只需在设置（Settings/Options）里点击 Download FFmpeg，它会自动搞定。

3. LosslessCut (手术刀：最快的无损视频切片与合并工具)

官方网站： https://losslesscut.app/（请认准此官方域名，谨防第三方下载站的捆绑垃圾软件）

这款工具基于 Electron 开发，跨平台表现极佳，且极其轻量，解压即用。

Windows 安装步骤

  1. 访问官网的 Download页面，往下滑找到 Windows 图标，下载 .zip压缩包（推荐便携版，不用安装）。
  2. 将下载的 .zip文件解压，直接得到一个名为 LosslessCut-win.exe的文件。
  3. 双击即可直接打开软件。*(注：因为是开源未签名软件，Windows Defender 可能会拦截，点击“更多信息” -> “仍要运行”即可)*。

macOS 安装步骤

  1. 前往官网，根据您的芯片下载对应的 .dmg文件（Intel 芯片选 x64，Apple Silicon 选 arm64）。
  2. 打开下载的 .dmg文件，将 LosslessCut.app拖入 Applications文件夹。
  3. 首次打开时，系统可能会提示“无法打开未经验证的开发者”。此时需要进入 系统设置 -> 隐私与安全性，往下划到安全性部分，点击仍然允许。

Linux 安装步骤

LosslessCut 为 Linux 提供了多种通用安装包（AppImage / Snap / Flatpak）：

  • 通用方法（AppImage，免安装）：

    1. 从官网下载 .AppImage文件。
    2. 右键文件 -> 属性 -> 权限，勾选“允许作为程序执行”。
    3. 双击即可运行。

  • 终端安装法（以 Ubuntu 为例，需先安装 Snapd）：

    sudo snap install losslesscut

当一个视频素材是由多段不同编码方式（例如一段是 H.264，另一段是 H.265，或者同一编码但分辨率/参数集不同）的视频拼接而成时，最头疼的问题就是全局的 Stream 信息往往只能反映第一段的编码参数。一旦后半段编码发生了变化，轻则播放器报错、花屏，重则直接导致合成后的视频无法播放。

为了帮您彻底摸清这个“缝合怪”的底细，我们需要放弃只看全局头信息的方法，转而对视频进行逐包（Packet）或逐帧（Frame）级别的“解剖”。以下是使用 FFmpeg/FFprobe 工具箱进行深度诊断的几个核心方案：

方案一：透视每一段数据的真实编码（逐 Packet 分析）

要找出视频内究竟夹杂了多少种编码格式，最有效的方法是让 ffprobe遍历每一个数据包（Packet），并按时间戳列出其真实的编码格式。

执行命令：

ffprobe -v error -show_packets -show_entries packet=pts_time,stream_index,codec_type,flags <input_file> 2>&1 | grep -E "PACKET|codec_type|stream_index|pts_time"

结果剖析：

  • stream_index: 如果视频中的某一段编码发生了变化（比如从 H.264 切到了 H.265），FFmpeg 通常会为其分配一个新的流索引（Stream 1, 2, 3…）。通过观察 stream_index的变化，您可以精确定位在哪段时间切换了编码。
  • flags: 如果是关键帧（K），通常伴随着新的 SPS/PPS 等参数集的刷新。

方案二：精准拦截并提取 SPS 和 PPS 参数集

SPS（序列参数集）和 PPS（图像参数集）是 H.264/H.265 等编码的“灵魂”，包含了分辨率、帧率、编码档次（Profile）等核心配置。对于拼接视频，我们需要确认每一段是否都携带了完整且正确的 SPS/PPS。

方法 1：使用 Bitstream Filter 强制“脱壳”提取

FFmpeg 提供了一个非常强大的比特流过滤器 dump_extra，可以直接把隐藏在视频流中的 SPS/PPS 等 extradata 打印到控制台。

针对 H.264 视频：

ffmpeg -i <input_file> -c:v copy -bsf:v dump_extra -f null - 2>&1

*(注：-bsf:v dump_extra会让 ffmpeg 在处理视频流时，把每个关键帧前面的 Extra Data 也就是 SPS/PPS 信息 dump 出来)*。

针对 H.265 / HEVC 视频：

ffmpeg -i <input_file> -c:v copy -bsf:v hevc_mp4toannexb -f null - 2>&1

*(注：HEVC 的参数是集成了 VPS/SPS/PPS 的，使用 hevc_mp4toannexb可以在转换过程中将其暴露出来)*。

方法 2：使用 ffprobe 监听 Debug 级别的 NALU 信息

如果您想看得更细致（比如具体到哪个 NALU 单元是 SPS，哪个是 PPS），可以利用 ffprobe的 debug 模式直接解析 H.264/H.265 的裸流结构。

步骤 1：先将 problematic 的视频提取为裸流（以 H.264 为例）

ffmpeg -i <input_file> -c:v copy -bsf:v h264_mp4toannexb -f h264 output.h264

*(注：MP4 等封装格式中，SPS/PPS 存在文件头，而裸流 NALU 格式要求每个关键帧前都有起始码和 SPS/PPS，h264_mp4toannexb过滤器就是干这个转换的)*。

步骤 2：用 ffprobe 深度解析裸流

ffprobe -v debug -show_frames -show_packets output.h264 2>&1 | grep -E "nal_type|SPS|PPS|IDR"

输出示例及含义：

nal_type=7           # NALU类型 7 代表 SPS (Sequence Parameter Set)
nal_type=8           # NALU类型 8 代表 PPS (Picture Parameter Set)
nal_type=5           # NALU类型 5 代表 IDR 帧 (关键帧)

如果在拼接处，您发现 nal_type=7或 nal_type=8再次出现，且里面的分辨率参数（如 pic_width_in_mbs_minus1）发生了变化，那就证明这里发生了编码参数的切换。

方案三：直捣黄龙（二进制层面验证 Extra Data）

有时候，所有的封装层工具都会骗人，这时候就需要上终极手段——直接看字节码。拼接视频最常见的问题就是 Extra Data（存储 SPS/PPS 的地方）在 MP4 的 avcCBox 中没有被正确更新。

执行命令：

ffprobe -v error -show_data -show_packets -select_streams v <input_file> 2>&1 | head -100

加上 -show_data后，FFprobe 会将每个 Packet 的十六进制数据直接打印出来。

  • H.264 的特征码：寻找以 67开头的十六进制行（代表 SPS 的 NALU 头），或者 68（代表 PPS）。

  • H.265 的特征符：寻找 40 01(VPS), 42 01(SPS), 44 01(PPS) 等 NALU 头。

如果在文件的中间部分突然冒出了新的 67或 68开头的数据包，说明这里的编码参数发生了变更。

方案四：借力打力（可视化辅助分析工具）

如果命令行下的输出让您感到眼花缭乱，这里推荐两个堪称神器的免费工具，它们本质上也是调用 FFmpeg 或类似底层库，但能提供可视化的时间轴分析：

  1. **MediaInfo (CLI或GUI版)**：

    运行 mediainfo --Details <input_file>，它可以非常清晰地按时间线列出视频的每一个关键帧以及对应的编码参数变化。

  2. FFmpeg Batch Converter / LosslessCut：

    这两款基于 FFmpeg 的图形化工具，在导入视频后，可以通过波形图和关键帧标记，直观地看到视频在哪些时间点发生了“跳跃”或参数变化，非常适合快速定位有问题的拼接点。

💡 避坑建议：如何修复这类视频？

排查完之后，如果您发现确实是因为多段不同编码直接拼接导致了 SPS/PPS 冲突或缺失，最彻底的修复方法是对发生变化的片段进行重新编码（Re-encode），而不是简单的流复制（Copy）。

例如，强制将整段视频统一转码为 H.264 High Profile：

ffmpeg -i <input_file> -c:v libx264 -profile:v high -crf 23 -c:a copy <output_fixed.mp4>

这样 FFmpeg 会在内部重新生成一套全局统一的 SPS/PPS，彻底消除因拼接带来的编码参数冲突隐患
